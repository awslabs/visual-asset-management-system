# VAMS v2.4 to v2.5 Migration Guide

## Overview

This migration copies data from V1 DynamoDB tables to new V2 tables with updated key schemas, and backfills a new field on the metadata versions table. The V2 tables use `databaseId`-prefixed composite keys that prevent collisions when an `assetId` is shared between databases and enable efficient database-scoped queries without GSIs.

### What This Migration Does

1. **Phase 1: Build Lookup Cache** - Scans the asset storage table to build an `assetId -> databaseId` mapping
2. **Phase 2: Migrate Asset Versions** - Full copy from V1 to V2 with transformed key schema
3. **Phase 3: Migrate Asset File Versions** - Full copy from V1 to V2 with transformed key schema
4. **Phase 4: Backfill Metadata Versions** - In-place update to add `databaseId:assetId` field for new GSI
5. **Phase 5: Verify** - Count records in V1/V2 tables, verify key structure, verify metadata backfill

### Table Key Schema Changes

| Table                        | V1 PK                                             | V1 SK                                     | V2 PK                               | V2 SK            |
| ---------------------------- | ------------------------------------------------- | ----------------------------------------- | ----------------------------------- | ---------------- |
| Asset Versions               | `assetId`                                         | `assetVersionId`                          | `databaseId:assetId`                | `assetVersionId` |
| Asset File Versions          | `assetId:assetVersionId`                          | `fileKey`                                 | `databaseId:assetId:assetVersionId` | `fileKey`        |
| Asset File Metadata Versions | _(unchanged)_ `databaseId:assetId:assetVersionId` | _(unchanged)_ `type:filePath:metadataKey` | —                                   | —                |

**Additional fields added:**

-   **Asset File Versions V2**: `databaseId:assetId` field (for `databaseIdAssetIdIndex` GSI)
-   **Asset File Metadata Versions**: `databaseId:assetId` field backfilled (for new `databaseIdAssetIdIndex` GSI)

### Why V2 Tables?

In V1, the `assetVersionsStorageTable` used `assetId` as the partition key. If the same `assetId` existed in multiple databases, their versions would collide. The V2 schema prefixes all partition keys with `databaseId:` to guarantee uniqueness across databases.

## Prerequisites

### Required AWS Resources

1. **v2.5 CDK Stack Deployed**: The V2 tables and updated GSIs must exist before running migration
2. **V1 Tables Still Exist**: The CDK stack keeps V1 tables (with their original CloudFormation logical IDs) as migration sources
3. **Table Names**: Available from CloudFormation stack outputs or AWS Console

## Usage

### Step 1: Find Table Names

Get table names from CloudFormation stack outputs:

```bash
aws cloudformation describe-stacks --stack-name your-vams-stack \
  --query 'Stacks[0].Outputs[?contains(OutputKey, `Table`)].{Key:OutputKey,Value:OutputValue}' \
  --output table
```

Or find them in the AWS Console under DynamoDB > Tables. Look for tables containing:

-   `AssetStorageTable` (lookup source)
-   `AssetVersionsStorageTable` (V1 source)
-   `AssetVersionsStorageTableV2` (V2 destination)
-   `AssetFileVersionsStorageTable` (V1 source)
-   `AssetFileVersionsStorageTableV2` (V2 destination)
-   `AssetFileMetadataVersionsStorageTable` (in-place backfill)

### Step 2: Configure Migration

Edit `v2.4_to_v2.5_migration_config.json`:

```json
{
    "asset_storage_table_name": "YOUR-STACK-AssetStorageTable-XXXXX",
    "asset_versions_storage_table_name_v1": "YOUR-STACK-AssetVersionsStorageTable-XXXXX",
    "asset_versions_storage_table_name_v2": "YOUR-STACK-AssetVersionsStorageTableV2-XXXXX",
    "asset_file_versions_storage_table_name_v1": "YOUR-STACK-AssetFileVersionsStorageTable-XXXXX",
    "asset_file_versions_storage_table_name_v2": "YOUR-STACK-AssetFileVersionsStorageTableV2-XXXXX",
    "asset_file_metadata_versions_storage_table_name": "YOUR-STACK-AssetFileMetadataVersionsStorageTable-XXXXX",
    "batch_size": 25,
    "dry_run": false,
    "limit": null,
    "aws_profile": null,
    "aws_region": "us-east-1",
    "log_level": "INFO"
}
```

### Step 3: Run Migration

#### Dry Run (Recommended First Step)

Test without making changes:

```bash
python v2.4_to_v2.5_migration.py \
    --config v2.4_to_v2.5_migration_config.json \
    --dry-run
```

#### Test with Limited Items

Test with a small subset:

```bash
python v2.4_to_v2.5_migration.py \
    --config v2.4_to_v2.5_migration_config.json \
    --limit 10 \
    --dry-run
```

#### Production Migration

Run the full migration:

```bash
python v2.4_to_v2.5_migration.py \
    --config v2.4_to_v2.5_migration_config.json
```

## Migration Process Details

### Phase 1: Build Lookup Cache

-   Scans the `AssetStorageTable` (PK=`databaseId`, SK=`assetId`)
-   Builds a dictionary mapping `assetId -> databaseId`
-   This cache is used in Phases 2-4 to resolve the `databaseId` for each asset

### Phase 2: Migrate Asset Versions (V1 -> V2)

For each record in V1 `AssetVersionsStorageTable`:

1. **Extract** `assetId` (V1 PK) and `assetVersionId` (V1 SK)
2. **Look up** `databaseId` from the Phase 1 cache
3. **Build V2 item** by copying all attributes and setting:
    - New PK: `databaseId:assetId` = `{databaseId}:{assetId}`
    - SK: `assetVersionId` (unchanged)
    - Fields: `databaseId`, `assetId` stored as separate attributes
4. **Batch write** to V2 table (batches of 25)
5. **Skip** orphaned records where `assetId` is not in the asset table

### Phase 3: Migrate Asset File Versions (V1 -> V2)

For each record in V1 `AssetFileVersionsStorageTable`:

1. **Parse** `assetId` and `assetVersionId` from V1 PK (`assetId:assetVersionId`)
2. **Look up** `databaseId` from cache
3. **Build V2 item** by copying all attributes and setting:
    - New PK: `databaseId:assetId:assetVersionId` = `{databaseId}:{assetId}:{assetVersionId}`
    - SK: `fileKey` (unchanged)
    - GSI field: `databaseId:assetId` = `{databaseId}:{assetId}`
    - Remove old V1 PK field `assetId:assetVersionId`
4. **Batch write** to V2 table (batches of 25)
5. **Skip** orphaned records

### Phase 4: Backfill Metadata Versions

The `AssetFileMetadataVersionsStorageTable` keeps its existing key schema (PK = `databaseId:assetId:assetVersionId`), but existing records need a new `databaseId:assetId` field for the new GSI.

For each record:

1. **Skip** if `databaseId:assetId` field already exists (idempotent)
2. **Derive** `databaseId:assetId` from the PK by stripping the last `:assetVersionId` segment
3. **Update** the record in-place with the new field

### Phase 5: Verification

-   Counts records in V1 and V2 tables for asset versions and file versions
-   Verifies V2 count >= V1 count minus orphaned records
-   Spot-checks V2 records for proper composite key structure
-   Verifies all metadata version records have the `databaseId:assetId` field

## Troubleshooting

### Common Issues

#### Table Not Found

**Symptom**: `Error scanning table: An error occurred (ResourceNotFoundException)`

**Solution**: Verify table names from CloudFormation outputs. Ensure you're using the correct AWS region and profile. The V2 tables are only created after deploying the v2.5 CDK stack.

#### Permission Errors

**Symptom**: `Access denied` or `User is not authorized`

**Solution**: Ensure IAM permissions include `dynamodb:Scan` on all source tables, `dynamodb:BatchWriteItem` on V2 tables, and `dynamodb:UpdateItem` on the metadata versions table.

#### Orphaned Versions

**Symptom**: Migration reports orphaned versions

**Explanation**: These are records whose `assetId` does not exist in the asset storage table. This can happen if:

-   An asset was deleted but its versions were not cleaned up
-   Data inconsistency from previous operations

**Action**: Orphaned records are skipped during migration and left in the V1 tables. They are harmless and can be investigated separately.

#### Timeout Issues

**Symptom**: Script hangs or times out on large tables

**Solution**:

1. Test with `--limit 100` first to verify connectivity
2. Ensure stable network connection to AWS
3. For very large tables (>10,000 records), consider running during low-traffic periods

### Rollback

This migration creates data in **new V2 tables** and adds a field to the metadata versions table. It does not modify or delete any V1 data.

-   **V1 tables remain intact** and can serve as rollback sources
-   **Re-running the migration is safe** (PutItem overwrites existing V2 records; backfill skips records that already have the field)
-   **To rollback**: Redeploy v2.4 CDK stack which points Lambda env vars back to V1 tables

## Performance Considerations

| Deployment Size | Total Records (across tables) | Estimated Duration     |
| --------------- | ----------------------------- | ---------------------- |
| Small           | < 500                         | Under 30 seconds       |
| Medium          | 500 - 5,000                   | 30 seconds - 3 minutes |
| Large           | 5,000 - 50,000                | 3 - 15 minutes         |
| Very Large      | > 50,000                      | 15+ minutes            |

Phases 2 and 3 use `batch_write_item` (25 items per batch) for efficient writes. Phase 4 uses individual `update_item` calls for in-place backfill. Unprocessed items from batch writes are retried up to 3 times.
