# VAMS v2.4 to v2.5 Migration Guide

## Overview

This migration backfills `databaseId` and `databaseId:assetId` fields on all existing asset version records in the `AssetVersionsStorageTable`. These fields are required for efficient querying of asset versions by database in VAMS v2.5.

### What This Migration Does

1. **Phase 1: Build Lookup Cache** - Scans the asset storage table to build an `assetId -> databaseId` mapping
2. **Phase 2: Backfill** - Scans the asset versions table and updates records that are missing `databaseId`
3. **Phase 3: Verify** - Counts records with/without `databaseId` and reports any orphaned versions

### What's Changed in v2.5

-   **Asset Versions Table**: Now includes `databaseId` and `databaseId:assetId` composite key fields
-   **New GSI**: `DatabaseIdAssetIdIndex` on the asset versions table enables efficient queries by database
-   **Existing records**: Need backfilling since they were created without these fields

```bash

  --query 'Stacks[0].Outputs[?contains(OutputKey, `Table`)].{Key:OutputKey,Value:OutputValue}' \
```

## Migration Process Details

### Phase 1: Build Lookup Cache

-   Scans the `AssetStorageTable` (PK=`databaseId`, SK=`assetId`)
-   Builds a dictionary mapping `assetId -> databaseId`
-   This cache is used in Phase 2 to look up the databaseId for each asset version

### Phase 2: Backfill databaseId

For each record in `AssetVersionsStorageTable`:

1. **Skip** if record already has `databaseId` set (idempotent)
2. **Look up** the `databaseId` from the Phase 1 cache using the record's `assetId`
3. **Update** the record with:
    - `databaseId`: The database the asset belongs to
    - `databaseId:assetId`: Composite key in format `{databaseId}:{assetId}`
4. **Report** orphaned versions where the `assetId` is not found in the asset table

### Phase 3: Verification

-   Scans the asset versions table one more time
-   Counts records with and without `databaseId`
-   Reports any orphaned versions (versions whose assetId has no matching asset)
-   Verification passes if all non-orphaned records have `databaseId`

## Troubleshooting

### Common Issues

#### Table Not Found

**Symptom**: `Error scanning asset storage table: An error occurred (ResourceNotFoundException)`

**Solution**: Verify table names from CloudFormation outputs. Ensure you're using the correct AWS region and profile.

#### Permission Errors

**Symptom**: `Access denied` or `User is not authorized`

**Solution**: Ensure IAM permissions include `dynamodb:Scan` on both tables and `dynamodb:UpdateItem` on the versions table.

#### Orphaned Versions

**Symptom**: Migration reports orphaned versions

**Explanation**: These are asset version records whose `assetId` does not exist in the asset storage table. This can happen if:

-   An asset was deleted but its versions were not cleaned up
-   Data inconsistency from previous operations

**Action**: Orphaned versions are harmless and can be investigated separately. The migration will complete successfully, leaving orphaned records unchanged.

#### Timeout Issues

**Symptom**: Script hangs or times out on large tables

**Solution**:

1. Test with `--limit 100` first to verify connectivity
2. Ensure stable network connection to AWS
3. For very large tables (>10,000 records), consider running during low-traffic periods

### Rollback

This migration only **adds** new fields (`databaseId` and `databaseId:assetId`) to existing records. It does not modify or delete any existing data. If you need to "rollback":

1. The added fields do not affect v2.4 functionality
2. Re-running the migration is safe (idempotent)
3. No manual rollback steps are needed

## Performance Considerations

| Deployment Size | Record Count          | Estimated Duration     |
| --------------- | --------------------- | ---------------------- |
| Small           | < 100 versions        | Under 30 seconds       |
| Medium          | 100-1,000 versions    | 30 seconds - 2 minutes |
| Large           | 1,000-10,000 versions | 2-10 minutes           |
| Very Large      | > 10,000 versions     | 10+ minutes            |

The migration uses individual `UpdateItem` calls (not batch writes) to ensure each record update is atomic and errors are isolated.
